# Issue 5 Follow-ups

This document tracks confirmed decisions and deferred review topics discovered
while shaping issue #5 (`Manage venues, courts, disciplines, and categories`).
The goal is to avoid losing context and later group related work into coherent
future issues instead of creating one issue per small pending item.

Deferred topics in this document are not automatic new issues. After the broader
issue plan is implemented or reviewed, each topic should be checked against the
existing/future backlog and then either removed, moved into an existing issue, or
grouped into a new issue only if it is still uncovered.

## Confirmed For Issue 5

- Keep one sidebar item: `Sedes e instalaciones`.
- Use one route with one active section at a time:
  - `/structure?section=venues`
  - `/structure?section=courts`
  - `/structure?section=disciplines`
  - `/structure?section=categories`
- If `section` is absent, `/structure` behaves like `venues`.
- There is no summary tab.
- Default section: `venues`.
- Each CRUD section has its own create action.
- Forms are inline inside the active section.
- Create forms open directly under the section header.
- Edit forms expand under the row; the original row remains visible.
- Tables default to active records only.
- Status filter is URL-backed:
  - `status=active`
  - `status=inactive`
  - `status=all`
- Status filter is a segmented control.
- Changing section resets filters to the section defaults.
- Courts also support a URL-backed venue filter:
  - `venue=all`
  - `venue={id}`
- The courts venue filter lists all venues, active and inactive.
- Creating a court lists only active venues.
- No local text search for now.
- No pagination for now.
- React Query is used for mutation state and can be expanded to list cache when the UI becomes more interactive.
- React Hook Form and Zod are used for forms.
- CRUD browser calls use relative `/api/...` paths.
- CSRF remains enabled:
  - read `csrftoken` from cookies
  - fetch `/api/auth/csrf/` if missing
  - send `X-CSRFToken` on mutations
  - keep `credentials: "include"`
- Backend list endpoints support query param filtering.
- Server Actions are used for the current form submissions.
- React Query provider is local to the academy structure surface for now.
- Tabs are compact visual tabs; on mobile, the tab row scrolls horizontally.
- Desktop uses compact tables.
- Mobile keeps the compact table surface with horizontal overflow for now.
- Row actions are visible on desktop, not hidden in a three-dot menu.
- Inactive rows are visually muted and still show an `Inactivo` badge.
- Users without admin/staff permissions do not see the sidebar item, but direct
  URL access shows a permission-denied state inside the shell.
- Admin/staff can manage this module for now; a future fine-grained permission
  such as `manage_academy_structure` can replace the role shortcut later.

## Confirmed Data Rules For Issue 5

- `Venue.name` is unique case-insensitively.
- `Court.name` is unique case-insensitively within the same venue.
- `Discipline.slug` is unique case-insensitively.
- `Category.name` is unique case-insensitively.
- `Category.code` is optional and unique case-insensitively when present.
- Uniqueness checks apply against active and inactive records.
- `Category.code` is visible in forms and tables.
- `Category.code` is manual, not autogenerated.
- `Category.code` is editable after creation.
- Empty `Category.code` is stored as `null`.
- Empty `Category.code` displays as `-` in tables.
- `Category.code` allows uppercase letters, numbers, and hyphens only.
- `Category.code` max length is 24.
- `Venue.notes` is optional, stored as `null` when empty, and visible as secondary table text.
- `Venue.notes` max length is 500 characters.
- `Court.notes` is optional, stored as `null` when empty, and only visible in edit/detail contexts.
- `Court.notes` max length is 500 characters.
- Notes allow line breaks, trim leading/trailing whitespace, and preserve internal formatting.
- `Venue.google_maps_url` is optional, stored as `null` when empty, and visible as an `Abrir mapa` link/action when present.
- `Venue.google_maps_url` must be a valid `https://` URL.
- `Venue.google_maps_url` opens in a new tab with `rel="noreferrer"`.
- `Venue` keeps the visible label `Sede`.
- `Venue` does not get contact fields yet.
- `Venue` gets optional `notes`.
- `Venue.name`, `Venue.address`, `Venue.district`, and `Venue.google_maps_url` are editable.
- `Court` does not get `surface_type` or `is_indoor` yet.
- `Court.venue` is selected at creation and is not editable afterward.
- If a court was created under the wrong venue, deactivate it and create a new court under the correct venue.
- `Court.name` and `Court.notes` are editable.
- `Discipline` keeps `sort_order` as an internal ordering field.
- `Discipline.slug` is autogenerated from name during creation, can be edited during creation, and is not editable after creation.
- `Discipline.slug` allows lowercase letters, numbers, and hyphens only.
- `Discipline.name` is editable. `Discipline.sort_order` is assigned automatically on create and is not shown in ordinary CRUD forms.
- Basketball and volleyball can be deactivated globally.
- `Category` age range rules:
  - both `min_age` and `max_age` empty is valid and does not participate in automatic suggestions
  - both present is valid
  - only one present is invalid
  - `min_age > max_age` is invalid
  - ages must be integers from 0 to 99
  - active category ranges cannot overlap other active categories
  - inactive category ranges do not block active ranges
  - editing an inactive category can overlap an active category
  - reactivating a category blocks if its range overlaps an active category
  - editing an active category blocks if its range overlaps another active category
  - age suggestion returns at most one category
  - staff can override the suggestion later in enrollment flows
- Multiple categories without ranges are allowed.
- Empty `min_age`/`max_age` displays as `Sin rango`.
- `Category.name` is editable.
- Deactivating a discipline or category does not break historical references.
- Inactive disciplines/categories should not be offered for new uses later.
- A venue cannot be deactivated while it has active courts.
- A venue can be deactivated if it only has inactive courts or no courts.
- Inactive records can be viewed, edited, and reactivated.
- Editing an inactive record shows an in-form inactive notice.
- Reactivating a court is blocked if its venue is inactive.
- Reactivating a venue does not reactivate its courts.
- Reactivation is a direct action without confirmation.
- Deactivation uses inline confirmation.
- `DELETE` remains supported as soft deactivate and is what the current frontend uses for deactivation.
- Current frontend deactivation uses the soft-delete endpoint; reactivation uses `PATCH { "status": "active" }`.
- Reactivation uses `PATCH { "status": "active" }`.
- Successful mutations return the updated resource.
- List responses keep the `{ "results": [...] }` shape.
- `name`, `address`, and `district` values are trimmed and have internal whitespace collapsed.
- `notes` values are trimmed only.
- `name` max length stays 120 for venues, courts, disciplines, and categories.
- `sort_order` accepts integers from 0 to 999 at the API/data layer, but ordinary CRUD UI omits it and lets the backend assign the next value.

## Deferred Follow-up Groups

### 1. Audit Log Foundation

Build shared audit logging before attaching audit history to individual modules.

Candidate scope:

- Add `AuditLog` model and write API/domain helpers.
- Record actor, action, entity type, entity id, before/after snapshots, timestamp, and request metadata where available.
- Add events for academy structure changes:
  - `venue.created`
  - `venue.updated`
  - `venue.deactivated`
  - `venue.reactivated`
  - `court.created`
  - `court.updated`
  - `court.deactivated`
  - `court.reactivated`
  - `discipline.created`
  - `discipline.updated`
  - `discipline.deactivated`
  - `discipline.reactivated`
  - `category.created`
  - `category.updated`
  - `category.deactivated`
  - `category.reactivated`
- Add a later `Configuracion > Auditoria` view.

Reason deferred:

- Audit applies across many modules: payments, scholarships, attendance overrides,
  receipt decisions, permissions, invitations, and academy structure.
- Implementing it only for issue #5 would create a partial system likely to be rewritten.
- This topic should be reviewed against future audit/security issues before
  creating a new issue.

### 2. District Catalog

Normalize venue districts into a catalog.

Candidate scope:

- Add `District` model with `name`, `status`, and `sort_order`.
- Seed common Lima districts if product confirms the source list.
- Migrate `Venue.district` from text to FK.
- Add district filter for venue tables/reports.
- Decide whether districts are limited to Lima or support custom entries.

Reason deferred:

- Current issue already has multiple structure CRUDs.
- District normalization is useful, but it is a broader cataloging concern.
- This is a review topic, not a guaranteed standalone issue.

### 3. Venue Discipline Availability

Represent which disciplines are offered at each venue.

Candidate scope:

- Add venue-discipline availability model.
- Allow staff/admin to configure that volleyball exists globally but is not offered at a specific venue.
- Use availability when creating class groups and schedules.
- Keep `Discipline.status` as global availability.

Reason deferred:

- Availability by venue affects groups, schedules, pricing, enrollment, and reporting.
- For issue #5, absence of a class group in a venue is enough to express that the discipline is not currently offered there.
- This may already belong in a future class groups, schedules, or enrollment issue.
  Review the backlog before creating anything new.

### 4. Class Groups And Schedules Integration

Use the academy structure when creating operational class groups and schedules.

Candidate scope:

- Create class groups from venue, court, discipline, category, gender mode, status, and capacity.
- Create schedules with weekday, start time, end time, date range, and status.
- Validate active venue/court/discipline/category for new records.
- Preserve historical references to inactive records.
- Add court conflict validation.
- Add coach conflict validation after coach assignments exist.
- Add override permissions later for authorized conflict overrides.

Reason deferred:

- Schedules are downstream of venues, courts, disciplines, categories, and class groups.
- Issue #5 should establish the base structure first.
- This may already be covered by later issues. Keep it here only as context to
  verify once those issues are implemented or planned.

### 5. Rich Form Error System

Standardize frontend and backend validation error codes.

Candidate scope:

- Backend returns stable error codes, not only final text.
- Frontend maps codes to Spanish/English messages.
- React Hook Form displays field-level errors.
- Zod handles immediate client-side validation.
- Backend remains authoritative for uniqueness and domain rules.

Example codes:

- `duplicate_name`
- `duplicate_slug`
- `duplicate_code`
- `age_range_pair_required`
- `age_range_invalid_order`
- `age_range_overlap`
- `active_courts_block_venue_deactivation`
- `inactive_venue_for_court`

Reason included:

- Issue #5 currently returns field-level API errors, while RHF/Zod handles the
  immediate client-side required-field path. Stable reusable error codes should
  be introduced as a shared convention before more modules depend on it.
